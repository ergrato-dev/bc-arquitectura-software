# 📊 Rúbrica de Evaluación — Semana 09: Proyecto Integrador Final

> **Sistema de evaluación**: Competencias laborales SENA
> **Criterio de aprobación**: Mínimo 70% en cada evidencia

---

## Distribución de Evaluación

| Evidencia       | Porcentaje | Descripción                                                                             |
| --------------- | ---------- | --------------------------------------------------------------------------------------- |
| 🧠 Conocimiento | 30%        | Síntesis de principios, patrones, calidad y justificación de decisiones arquitectónicas |
| 💪 Desempeño    | 40%        | Presentación técnica, defensa ante preguntas, documentación ADRs y diagramas C4         |
| 📦 Producto     | 30%        | Sistema funcional completo: 8 semanas integradas, documentado y desplegable             |

---

## 🧠 Evidencia de Conocimiento (30%)

_Evalúa la comprensión e integración de todos los conceptos del bootcamp._

### Criterio 1: Síntesis de Principios Arquitectónicos

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                                                  |
| ---------------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Relaciona con precisión los 5 pilares del bootcamp: SOLID, patrones clásicos, arquitectura hexagonal/clean, cloud y seguridad. Explica cómo cada semana construye sobre la anterior con ejemplos específicos de su proyecto. |
| **Bueno**        | 85%        | Relaciona al menos 4 de los 5 pilares y puede explicar las dependencias entre semanas. Los ejemplos son concretos aunque alguna relación queda implícita.                                                                    |
| **Aceptable**    | 70%        | Puede describir cada semana individualmente y menciona al menos 2 relaciones entre ellas. Comprende la idea de arquitectura evolutiva aunque no la articula con fluidez.                                                     |
| **Insuficiente** | <70%       | Describe las semanas como temas aislados. No puede explicar cómo un concepto aprendido impacta a los subsecuentes.                                                                                                           |

### Criterio 2: Justificación de Decisiones Arquitectónicas

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                    |
| ---------------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Para cada decisión clave (patrón seleccionado, tecnología elegida, trade-off aceptado), presenta alternativas consideradas y criterios de selección basados en atributos de calidad ISO 25010. |
| **Bueno**        | 85%        | Justifica la mayoría de decisiones con alternativas y criterios. Puede faltar profundidad en una o dos decisiones, pero el razonamiento es técnicamente sólido.                                |
| **Aceptable**    | 70%        | Justifica las decisiones más importantes. Las justificaciones son válidas aunque simples. Puede no haber considerado alternativas explícitamente.                                              |
| **Insuficiente** | <70%       | Las decisiones se justifican con "lo aprendí en clase" o preferencia personal sin criterios técnicos.                                                                                          |

### Criterio 3: Atributos de Calidad y Trade-offs

| Nivel            | Porcentaje | Descripción                                                                                                                                                                  |
| ---------------- | ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Identifica al menos 4 atributos de calidad ISO 25010 relevantes para su sistema. Describe 2+ trade-offs aceptados con su justificación y las consecuencias asumidas.         |
| **Bueno**        | 85%        | Identifica 3 atributos de calidad y describe 1-2 trade-offs. La relación entre atributos y decisiones de diseño es clara.                                                    |
| **Aceptable**    | 70%        | Conoce el concepto de atributos de calidad y puede mencionar al menos 2 (ej. mantenibilidad, seguridad). Reconoce que existen trade-offs aunque no los describe con detalle. |
| **Insuficiente** | <70%       | No puede explicar qué es un atributo de calidad o no puede identificar ninguno en su sistema.                                                                                |

### Criterio 4: Documentación con ADRs

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                  |
| ---------------- | ---------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Entrega al menos 3 ADRs bien formados: título descriptivo, contexto suficiente, opciones consideradas, decisión tomada, consecuencias (positivas y negativas) y estado (aceptado/deprecado). |
| **Bueno**        | 85%        | Entrega 3 ADRs con los campos principales completos. Puede faltar profundidad en "consecuencias" o no describir las alternativas descartadas en detalle.                                     |
| **Aceptable**    | 70%        | Entrega 2 ADRs con al menos contexto, decisión y consecuencias básicas.                                                                                                                      |
| **Insuficiente** | <70%       | No entrega ADRs o los entrega incompletos (solo título y decisión sin contexto ni consecuencias).                                                                                            |

## 💪 Evidencia de Desempeño (40%)

_Evalúa la capacidad de presentar, defender y documentar el sistema con diagramas._

### Criterio 1: Presentación Técnica (7 minutos)

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                           |
| ---------------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Estructura clara: contexto del dominio → problema resuelto → decisiones arquitectónicas clave → demo en vivo → trade-offs. Rango de tiempo respetado (6-8 min). Vocabulario técnico preciso y fluido. |
| **Bueno**        | 85%        | Cubre los puntos principales con claridad. La demo funciona correctamente. Puede extenderse ligeramente o le falta profundidad en un bloque, pero la presentación es coherente.                       |
| **Aceptable**    | 70%        | Presenta el sistema aunque con poca estructura. La demo puede tener problemas menores. El vocabulario técnico es básico pero suficiente para comunicar las ideas.                                     |
| **Insuficiente** | <70%       | La presentación carece de estructura visible, el demo falla o no puede explicar las decisiones tomadas con argumentos técnicos.                                                                       |

### Criterio 2: Defensa ante Preguntas (Q&A — 3 minutos)

| Nivel            | Porcentaje | Descripción                                                                                                                                                           |
| ---------------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Responde preguntas técnicas con argumentos precisos. Si no sabe una respuesta, lo admite y propone cómo investigaría. Puede sugerir mejoras al sistema en el momento. |
| **Bueno**        | 85%        | Responde la mayoría de preguntas con fundamento. Alguna respuesta puede ser superficial pero demuestra comprensión general del sistema.                               |
| **Aceptable**    | 70%        | Responde preguntas básicas sobre su sistema. Tiene dificultades con preguntas sobre alternativas o sobre decisiones específicas.                                      |
| **Insuficiente** | <70%       | No puede responder preguntas sobre su propio diseño o las responde con información incorrecta.                                                                        |

### Criterio 3: Diagramas C4 Model (Nivel 1 y 2)

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                                     |
| ---------------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Diagrama de Contexto (C4 L1): sistema, actores y sistemas externos claramente identificados. Diagrama de Contenedores (C4 L2): todos los contenedores principales con tecnología, protocolos y flujos de datos. |
| **Bueno**        | 85%        | Ambos diagramas presentes y correctos en estructura. Puede faltar algún sistema externo en L1 o algún protocolo explícito en L2, pero la lectura del diagrama es clara.                                         |
| **Aceptable**    | 70%        | Al menos uno de los diagramas (C4 L1 o L2) completo y correcto. El otro puede estar incompleto o confundir niveles.                                                                                             |
| **Insuficiente** | <70%       | No entrega diagramas C4 o entrega uno que no sigue la notación del modelo (p.ej. confunde con casos de uso o entidad-relación).                                                                                 |

### Criterio 4: Retroalimentación a Par (Peer Review)

| Nivel            | Porcentaje | Descripción                                                                                                                                                                     |
| ---------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Provee retroalimentación técnica a al menos un compañero: identifica al menos una fortaleza arquitectónica concreta y una área de mejora con propuesta alternativa justificada. |
| **Bueno**        | 85%        | Provee retroalimentación técnica con fortaleza y área de mejora. La mejora está identificada pero sin propuesta alternativa concreta.                                           |
| **Aceptable**    | 70%        | Provee retroalimentación básica aunque centrada en aspectos superficiales (presentación, código limpio) más que en decisiones arquitectónicas.                                  |
| **Insuficiente** | <70%       | No participa en el peer review o la retroalimentación es exclusivamente positiva sin análisis crítico técnico.                                                                  |

## 📦 Evidencia de Producto (30%)

_Evalúa el sistema entregado en repositorio GitHub: completitud, funcionamiento y documentación._

### Criterio 1: Sistema Funcional Completo (Semanas 1–8)

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                |
| ---------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Excelente**    | 100%       | La API del dominio asignado integra: arquitectura hexagonal (puertos/adaptadores), al menos 2 patrones GoF, containerización Docker + Compose, y capa de seguridad JWT + RBAC + hardening. |
| **Bueno**        | 85%        | La API integra arquitectura hexagonal, patrones GoF y Docker. La capa de seguridad puede estar parcialmente implementada (JWT sin RBAC o sin hardening completo).                          |
| **Aceptable**    | 70%        | La API tiene arquitectura hexagonal o limpia funcional y al menos un patrón GoF. Falta containerización o seguridad pero el núcleo del dominio funciona.                                   |
| **Insuficiente** | <70%       | La API no tiene separación de capas clara, no aplica patrones GoF o el sistema no inicia correctamente.                                                                                    |

### Criterio 2: Repositorio GitHub Organizado

| Nivel            | Porcentaje | Descripción                                                                                                                                                                         |
| ---------------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | Historial de commits con mensajes descriptivos (conventional commits o similar). Estructura de carpetas coherente con la arquitectura. `.gitignore` correcto. `README.md` completo. |
| **Bueno**        | 85%        | Estructura organizada y `README.md` con instrucciones de instalación. Los commits pueden ser menos descriptivos pero el repositorio refleja el progreso semanal.                    |
| **Aceptable**    | 70%        | Repositorio con el código y un `README.md` básico de instalación. La historia de commits puede estar desordenada.                                                                   |
| **Insuficiente** | <70%       | No tiene repositorio público, o el repositorio tiene todo el código en un solo commit, o no tiene `README.md`.                                                                      |

### Criterio 3: Documentación Técnica Completa

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                |
| ---------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Excelente**    | 100%       | Entrega: README con guía de instalación y ejecución, 3+ ADRs bien formados, diagramas C4 (L1 + L2), colección Postman/Insomnia o equivalente, y reporte de seguridad (hardening aplicado). |
| **Bueno**        | 85%        | Entrega README, 2-3 ADRs y diagramas C4 (al menos L1 + L2). Puede faltar la colección de API o el reporte de seguridad.                                                                    |
| **Aceptable**    | 70%        | Entrega README con instrucciones básicas y al menos 2 ADRs. Los diagramas pueden ser un único nivel C4 (solo contexto o solo contenedores).                                                |
| **Insuficiente** | <70%       | Solo entrega el código sin documentación adicional, o la documentación entregada no corresponde al sistema implementado.                                                                   |

### Criterio 4: `.env.example` y Despliegue Reproducible

| Nivel            | Porcentaje | Descripción                                                                                                                                                                                       |
| ---------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Excelente**    | 100%       | El sistema levanta con `docker compose up` sin modificar archivos. `.env.example` tiene todas las variables necesarias documentadas. No hay secretos en el repositorio. `pnpm test` pasa al 100%. |
| **Bueno**        | 85%        | El sistema levanta con Docker Compose y pasos mínimos documentados. `.env.example` está presente con las variables principales. Los tests pasan.                                                  |
| **Aceptable**    | 70%        | El sistema levanta con instrucciones en el README (puede requerir pasos manuales). Variables de entorno documentadas aunque incompletas.                                                          |
| **Insuficiente** | <70%       | El sistema no puede levantarse sin asistencia del autor, o tiene secretos hardcodeados en el código fuente/repositorio.                                                                           |
